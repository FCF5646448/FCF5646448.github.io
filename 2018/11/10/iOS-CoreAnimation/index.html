iOS-CoreAnimation

发表于 2018-11-10

CoreAnimation: 不止于动画。动画只是CoreAnimation的冰山一角而已，UI的渲染也都是靠CoreAnimation。

1、图层&视图

图层 CALayer：只能做视图处理

视图 UIView: 能够响应事件

CALayer和UIView最大的不同是UILayer不处理用户交互。但它们是一种平行的层级关系，每一个视图UIView都含有一个UILayer，一个视图View可以添加多个Layer。苹果之所以这样设计是为了达到一种职责分离的效果。

4、视觉效果

CALayer提供了几个很好的属性来实现视觉效果

  • 圆角：conrnerRadius，用来设置圆角曲率的值。一般与maskToBounds结合使用，将圆角之外的内容裁剪掉。但是如果要实现部分圆角部分直角的视图，则需要使用其他的办法，比如图层蒙版或者专用图层CAShaperLayer
  • 边框：border，包括width和color
  • 阴影：shadow，阴影继承自内容的外形。

阴影裁剪：

阴影裁剪是开发过程中经常碰到的问题，因为阴影本身是处在图层边界之外的，而视图裁剪开启了maskToBounds会把裁剪之外的额内容都去掉。比如实现一个圆角带阴影的视图，那么实现的方式就只能是用两个图层来实现:一个化应用的外图层和一个用maskToBounds的内图层

阴影性能优化：在设置阴影的时候，iOS会先绘制目标的阴影，然后绘制目标本身，在没有指定阴影路径的时候，iOS在视图每次绘制前都会递归地精确计算每个子层的阴影，这也就造成了离屏渲染的问题。而添加了shadowpath就相当于告诉了GPU每次需要绘制的路径，这样就解决了离屏渲染的问题。接到上面代码，添加：

5、性能优化
  • CPU VS GPU

可以说CPU主要负责软件层面的工作，GPU主要负责硬件层面的工作。
动画和屏幕上组合的图层实际上被一个单独的进程管理，而不是你的应用程序。这个进程就是所谓的渲染服务。
当运行一个动画时，core animation如何为CPU和GPU分配工作呢？
举个例子：当一个动作发生时，会被分解成这几个部分：

布局：即视图/图层层级关系，以及设计图层的属性（位置、背景色、边框）
显示：绘制阶段，可能涉及drawRect和drawLayer:inContext:方法里。
准备：图片编解码
提交：core animation打包所有图层和动画属性，通过IPC发送到渲染服务进行显示
显示1：对所有图层计算中间值，设置OpenGL几何形状来执行渲染
显示2：在屏幕上渲染可见的三角形

这里前面5个阶段由CPU进行处理。最后阶段由GPU执行。

  • GPU：它用来采集图片和形状，运行变换，应用纹理和混合然后把它们输送到屏幕上。但是下面的操作会降低GPU图层绘制。主要是通过影响帧率。

    • 太多几何结构；
    • 重绘；
    • 离屏绘制；
    • 过大的图片

  • CPU：CPU的操作都发生在显示之前，所以不会影响到帧率。但是有些因素会延迟绘画开始时间，使界面看起来迟钝。

    • 布局计算；

    • 视图懒加载；（懒加载对视图启动时间很好，但是懒加载会比正常的操作慢一些）

    • Core Graphics绘制；解压图片；

  • IO操作： IO操作是指闪存或者网络接口的硬件访问（比如远程URL加载图片，加载nib文件）。IO比内存访问更慢。

    6、高效绘图

  • 脏区域：需要重绘的区域称为脏区域。普通的绘图，当一个视图被改动过一小部分，重绘则会将整个寄宿图都重绘，如果使用setNeedsDisplayInRect:指定需要重绘的区域，则可以减少重绘的内容。

  • 异步绘制：CATiledLayer和drawsAsynchronously是绘制可以放到另外一个线程进行，然后将绘制的内容直接设置为图层。

7、图像IO
  • 加载与潜伏：图片需要消耗很大一部分内存，而且不太可能把需要显示的图片一次性放入内存中，所以需要周期性地加载和卸载图片。其次图片加载的速度被CPU和IO同时影响。其次图片的数量(比如快速滑动cell)和远程加载都将造成等待。所以需要引入后台线程。
    • 线程加载：使用GCD或NSOperation加载图片，然后在主线程中设置。
    • 延迟解压：一旦图片被加载就必须进行解码，解码过程需要消耗非常长的时间和大量内存。避免延时解压的方法就是使用imageName:方法，或者绕过UIKit，使用ImageIO（CGImage ）强制图片立刻解压，或者使用。
    • CATileLayer: 异步加载大型图片。
  • 缓存：滑动显示，最后不要一次性加载多张图片，而应该选择性地缓存。imageNamed：自动缓存了解压后的图片，但是最好不要用它加载大图，有可能被移除。
8、图层性能

  • 隐式绘制：使用特性的图层属性；使用特定的图层子类。比如用CAATextLayer代替UILabel。不要在内容不断变化的图层上使用光栅化，否则光栅化缓存方面的好处将会消失，而且让性能变得更糟。

  • 离屏渲染：图层的以下属性将会触发离屏绘制

    • 圆角（当和maskToBounds一起使用时）
    • 图层蒙板
    • 阴影
      对于那些需要动画而且要在屏幕外渲染的图层来说，可以使用CAShapeLayer、contentsCenter、shadowPath替代。
      圆角：使用CAShapeLayer+UIBezierPath 的 bezierPathWithARoundedRect:cornerARadius:代替。最后的方式是使用一个圆角背景图片。

  • 混合和过度绘制：合并不同图层的透明重叠像素消耗的资源也是很客观的。明智地使用shouldRasterize属性。将固定的图层体系折叠成单张图片，这样就不需要每一帧重新合成，也不会因为子图层导致混合和过度绘制了。

  • 减少图层数量：使用缓存复用，减少图层的创建。类似cell
